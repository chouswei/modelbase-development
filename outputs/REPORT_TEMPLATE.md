# Report template (implement side)

The **implement report** (alignment, gaps, traceability) is **generated by the implement side** and delivered to the modelbase. **One report = one iteration delivery.** Every report must include the **iteration block** and updated alignment, gaps, and traceability for what was done in that iteration. Only the **modelbase** decides whether the iteration is done; the implement side only gives the report.

**Per domain the implement side may be named differently:** e.g. **codebase** (software), **hardware repo**, **mechanism repo**, **deployment repo**.

---

## How to use (per iteration)

1. **Use** the structure below (iteration block, alignment, gaps, traceability).
2. **Produce one report per iteration.** Use one combined doc or three docs: **MODELBASE_ALIGNMENT.md**, **GAPS.md**, **IMPLEMENTATION_PLAN.md** (in the implement-side repo `docs/`).
3. **Every report:** At the top, fill the **iteration block** (iteration id, plan/model ref, scope of this delivery, changes since last report, suggested modelbase actions). Then update alignment, gaps, and traceability for the steps/phases covered in this iteration.

---

## Start of iteration (implement side checklist)

Before starting work for this iteration:

- [ ] **Plan:** I have the **current** development plan from the modelbase (latest version after the last iteration boundary). If the modelbase updated the plan after the last report, use that version.
- [ ] **Scope:** I know which phases or steps are in scope for this iteration (from the plan’s phases and checkable steps, and any "after v1" items the modelbase promoted).
- [ ] **Model:** I have the current model (requirements, deploy, behaviour) the plan refers to.

Then: implement (step 2), integrate and verify (step 3), produce report (step 4). One iteration ends when you deliver the report to the modelbase.

---

## Iteration block (every report)

Include this at the top of each report. It identifies **which iteration** this report is for and gives the modelbase the input it needs to **decide** (only modelbase decides if the iteration is done) and to update the plan for the next iteration.

| Field | What to put |
|-------|-------------|
| **Iteration** | Iteration number or label (e.g. *Iteration 1*, *Iteration 2*, *v1 delivery*, *Report 2025-03*). |
| **Plan / model ref** | Link or path to the **exact** development plan and model this report is against (e.g. modelbase `outputs/<project>-DEVELOPMENT_PLAN.md`, or **commit/tag** so the modelbase knows which plan version you used). |
| **Scope of this delivery** | Which phases or steps this report covers (e.g. Phase 1–2 done; Phase 3 in progress). Matches what was in scope for this iteration. |
| **Changes since last report** | What changed in implementation or scope since the previous report. *(First iteration: "N/A – first report.")* |
| **Suggested modelbase actions** | What you suggest the modelbase do (e.g. *Accept phase 1–2; update plan for phase 3 / next iteration*, *Update model: add optional action X*, *Close plan*). The **modelbase** decides; this field is input to that decision. |

---

## End of iteration (implement side checklist)

Before delivering the report to the modelbase:

- [ ] **Iteration block** filled (iteration id, plan ref, scope, changes since last report, suggested modelbase actions).
- [ ] **Alignment** updated for the steps/phases covered in this iteration; note what was fixed since last report.
- [ ] **Gaps** refreshed (add new, remove or mark resolved); behaviour/pipeline alignment noted.
- [ ] **Traceability** table extended for new steps; existing rows kept.
- [ ] Report delivered to modelbase (e.g. PR, email, or doc link). Modelbase will then decide if iteration is done and update the plan for the next iteration.

---

## When to produce the next report (next iteration)

- **End of this iteration:** You have implemented the scope for this iteration (phases/steps), integrated and verified. Produce and deliver the report. The modelbase decides if done and updates the plan; you then start the **next** iteration with the updated plan (or continue if not done).
- **Start of next iteration:** After the modelbase has updated the plan, pull the new plan and start step 2 (Implement) for the new scope. When that work is done, produce the **next** report (next iteration number).
- **Milestone or "after v1":** When the modelbase promotes §6 items into scope, your next iteration implements those; the report for that iteration references the updated plan.

---

## Report parts

| Part | Purpose | Per iteration |
|------|---------|----------------|
| **Iteration block** | Identifies this delivery; gives modelbase input to decide and update plan | **Mandatory** at top of every report. |
| **Alignment** | Implementation vs plan and model | Update for any new or changed steps/phases; note what was fixed since last report. |
| **Gaps** | Model vs implementation; limitations | Refresh list; add new gaps, remove or mark resolved ones. |
| **Traceability** | Plan step → implementation | Extend table for new steps; keep existing rows. |
