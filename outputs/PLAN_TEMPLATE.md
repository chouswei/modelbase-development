# Plan template (modelbase)

The **development plan** (plan to do) is **generated by the modelbase** and is the implementation guide for the **implement side**. The plan is **updated at iteration boundaries** when the modelbase receives an implement report, decides the iteration is done, and adjusts scope, phases, or steps for the next iteration.

---

## How to use

1. **Copy** the plan template structure into your project’s `outputs/` as `outputs/<project>-DEVELOPMENT_PLAN.md` (see **Template contents** below).
2. **Fill** placeholders: `<Project name>`, design links, tech stack, scope, phases, checkable steps, verification, references. Keep §0 (Modelbase ↔ implement side workflow).
3. **Point Gaps** to the implement-side repo (e.g. `outputs/<project>-GAPS.md` → "gaps maintained in implement-side repo at …"). Per domain the implement side may be named e.g. codebase (software), hardware repo, mechanism repo, deployment repo.
4. **At each iteration boundary:** Only the **modelbase** decides when an iteration is done (after reviewing the implement report). When done, modelbase **updates this plan** for the next iteration (scope, phases, §6 backlog, or close). §6 (After v1) is the backlog for the next iteration once the current milestone is done.

---

## Iteration and the plan

### Plan versioning (required)

At the top of the plan (or in §7 References), **always** record:

- **Plan version / iteration:** e.g. "Valid for iteration 3" or "Updated 2025-03 after iteration 2 report". *(First version: e.g. "Initial" or "Iteration 1".)*
- **Change note:** One line on what changed from the previous version (e.g. "Promoted §6 items X, Y into Phase 4; closed phase 3."). *(First version: "Initial plan.")*

This is required so the implement side and reports can reference the exact plan version, and the modelbase can track what changed between iterations.

### When to update the plan

| When | What to do with the plan |
|------|---------------------------|
| **Before iteration 1** | Publish initial plan (phases, steps). Set **Plan version** (e.g. "Initial" or "Iteration 1") and **change note** (e.g. "Initial plan."). Plan flows to implement side. |
| **After each implement report** | **Modelbase decides** whether this iteration is done. If **done:** modelbase **updates the plan** for the next iteration: set new scope, promote §6 After v1 into phases/steps if needed, or close; **update Plan version** and **change note**. If **not done:** plan may still be updated (e.g. clarifications); if updated, bump version and add change note; implement side continues with same or updated plan. |
| **Start of next iteration** | Implement side uses the **current** plan (and model). The updated plan is the contract for the next iteration. Ensure the next report’s "Plan / model ref" points to this version (path, commit, or tag). |
| **Milestone done** | Promote "After v1" items (§6) into the main phases/steps for the next iteration, or close the plan; update Plan version and change note. |

The plan is the **single reference** for what the implement side should build in the **current** iteration; keep it aligned with the model and with what was agreed after the last report.

### Modelbase checklist at iteration boundary (after report)

- [ ] Review implement report (alignment, gaps, traceability, suggested actions).
- [ ] Decide: is this iteration done? (Only modelbase decides.)
- [ ] If done: update plan for next iteration (scope, §6, or close); **update Plan version and change note** (required).
- [ ] If not done: update plan only if clarifications or small changes needed; if updated, **update Plan version and change note** (required).
- [ ] Implement side can then start the next iteration (or continue) with the current plan.
